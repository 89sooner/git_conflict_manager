# Embedded Systems & SoC Code Modification Rules

이 규칙은 모든 에이전트가 PR 분석, 코드 리뷰, 충돌 해결 가이드를 생성할 때 최우선으로 적용되어야 합니다.

1. **하드웨어 종속 파일 감시:** - 파일 확장자가 `.dts`, `.dtsi` (Device Tree), 또는 `register_map`을 포함하는 파일이 수정된 경우, 이를 단순 코드 변경으로 취급하지 마세요.
   - 반드시 "하드웨어 레지스터 맵 또는 디바이스 트리 변경이 감지되었습니다. 타깃 보드의 부팅 장애를 유발할 수 있으므로 BSP/Architect 팀의 승인이 필수입니다."라는 경고문을 출력해야 합니다.
2. **생성 코드(Generated Code) 분리:**
   - 빌드 과정에서 스크립트에 의해 자동 생성되는 파일 헤더나 C 파일 수정이 감지되면, 직접 병합(Merge)을 유도하지 마세요.
   - "이 파일은 자동 생성물입니다. 소스 스크립트(또는 설정 파일)를 수정한 후 빌드 시스템을 통해 재생성하세요"라고 안내해야 합니다.
3. **바이너리/아티팩트 배제:**
   - `.hex`, `.bin`, `.elf` 등의 파일 사이즈 변경을 텍스트 diff 로 분석하려고 시도하지 마세요.
