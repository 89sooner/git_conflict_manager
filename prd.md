# GitHub Enterprise 전환 지원 웹 도구 PRD

## 1. 문서 개요

* 문서명: GitHub Enterprise 전환 지원 웹 도구 PRD
* 작성 목적: Perforce 기반 대규모 임베디드 개발 조직이 GitHub Enterprise로 전환할 때 발생하는 사용 미숙, 브랜치 전략 혼선, 병합 충돌, 품질 저하, 운영 리스크를 줄이기 위한 웹 기반 지원 도구의 제품 요구사항 정의
* 대상 조직: 약 400명 규모의 SoC/임베디드 소프트웨어 개발 조직
* 제품 성격: GitHub Enterprise 보완형 운영 도구
* 배포 형태: 사내 웹 애플리케이션 + GitHub App + CI 연동 서비스

## 2. 배경 및 문제 정의

### 2.1 현재 상황

조직은 오랜 기간 Perforce를 사용해 반도체 SoC 및 임베디드 소프트웨어 소스코드를 관리해왔다. 기존 개발자들은 changelist 중심의 작업 방식, 중앙집중형 승인 흐름, 브랜치 사용 최소화, 파일 잠금 및 보수적 병합 전략에 익숙하다. 반면 Git/GitHub는 분산형 버전 관리, 브랜치 기반 협업, pull request 중심 리뷰, rebase/cherry-pick/merge 등의 개념을 요구한다.

### 2.2 핵심 문제

GitHub Enterprise 도입 이후 다음 문제가 동시에 발생할 가능성이 높다.

1. 개발자가 HEAD, branch, merge, rebase, cherry-pick, reset, reflog 등의 개념을 정확히 이해하지 못해 히스토리를 손상시키거나 잘못된 병합을 수행한다.
2. Perforce의 changelist 사고방식이 Git commit/branch/PR 모델로 자연스럽게 전환되지 않아 작업 단위가 불명확해진다.
3. 대규모 임베디드 저장소에서 장시간 빌드, 다중 타깃, 생성 코드, 서브모듈, 릴리스 브랜치 관리 등으로 인해 단순 Git 교육만으로는 품질을 보장하기 어렵다.
4. GitHub의 원시 기능만으로는 초급 사용자에게 현재 상태와 다음 행동을 충분히 안내하기 어렵다.
5. 코드 리뷰, 품질 게이트, 소유권 승인, 병합 정책이 저장소마다 다르게 설정되면 운영 표준이 무너진다.
6. 개발자 실수로 force push, 잘못된 base branch 선택, merge queue 미사용, 리뷰 누락, 필수 체크 미통과 상태 병합 시도가 증가한다.
7. 운영 조직은 전체 저장소의 전환 성숙도, 위험 PR, 반복 충돌 구간, 병목 리뷰 팀, 브랜치 정책 위반을 한눈에 파악하기 어렵다.

### 2.3 제품 기회

GitHub Enterprise 자체 기능을 표준 정책 엔진으로 사용하고, 그 위에 조직 맞춤형 안내/진단/품질 가이드를 얹는 웹 도구를 제공하면 전환 초기의 학습 비용과 품질 리스크를 동시에 줄일 수 있다.

## 3. 제품 비전

개발자가 Git 내부 개념을 몰라도 안전하게 작업할 수 있게 하고, 숙련자는 더 빠르게 협업할 수 있게 하며, 운영자는 조직 전체의 저장소 품질과 협업 흐름을 정책 기반으로 통제할 수 있는 전환 지원 플랫폼을 만든다.

## 4. 제품 목표

### 4.1 비즈니스 목표

1. GitHub Enterprise 전환 후 6개월 내 병합 실패율과 잘못된 브랜치 작업으로 인한 복구 건수를 유의미하게 줄인다.
2. 조직 전체의 PR 기반 협업 표준을 정착시킨다.
3. 코드 리뷰, 품질 체크, 소유권 승인, 릴리스 브랜치 정책을 중앙 정책으로 관리한다.
4. 임베디드/SoC 특성에 맞는 품질 게이트와 변경 위험 분석을 표준화한다.

### 4.2 사용자 목표

1. 개발자는 현재 Git 상태를 직관적으로 이해하고 다음 행동을 추천받는다.
2. 개발자는 merge/rebase/cherry-pick/conflict 상황에서 실수 없이 복구하거나 진행한다.
3. 리뷰어는 PR의 위험도와 영향 범위를 빠르게 파악한다.
4. 팀 리더와 품질 조직은 저장소/브랜치/PR 상태를 대시보드로 모니터링한다.

### 4.3 성공 지표

1. PR 최초 승인까지 평균 시간
2. PR merge lead time
3. merge conflict 발생률
4. rebase/cherry-pick 실패 후 지원 요청 건수
5. 필수 체크 미통과 PR 비율
6. 코드 소유자 승인 누락 비율
7. 강제 push 및 정책 우회 건수
8. 릴리스 브랜치 품질 게이트 실패 비율
9. 도구 월간 활성 사용자 비율
10. 신규 개발자 온보딩 완료 시간

## 5. 비목표

1. GitHub 자체를 대체하는 전체 SCM 플랫폼을 만들지 않는다.
2. 사용자의 로컬 Git 클라이언트를 완전히 대체하지 않는다.
3. 범용 IDE를 만들지 않는다.
4. 모든 병합 충돌을 자동 해결하겠다고 약속하지 않는다.
5. 임베디드 빌드 시스템 전체를 본 제품 하나로 통합하지 않는다.

## 6. 대상 사용자

### 6.1 1차 사용자

* 일반 개발자: Git 사용 미숙, 일상적인 commit/branch/PR/merge 작업 수행
* 숙련 개발자: 복잡한 rebase, cherry-pick, backport, release 관리 수행
* 리뷰어: 코드 리뷰, 승인, 리스크 판단 수행
* 팀 리더: 브랜치 정책, 리뷰 SLA, 병목 관리
* 품질/플랫폼 조직: 전사 정책 운영, 리포지토리 거버넌스, 감사

### 6.2 사용자 페르소나

#### 페르소나 A: Perforce 숙련 임베디드 개발자

* changelist 중심 사고에 익숙함
* PR, rebase, local branch 작업이 불편함
* 히스토리 재작성에 대한 두려움이 큼
* 원하는 것: 실수 없는 가이드와 안전장치

#### 페르소나 B: Git 숙련 모듈 리드

* 다수 브랜치/릴리스 관리 가능
* 팀원의 실수 복구와 리뷰 병목이 부담
* 원하는 것: 품질 자동화와 정책 일관성

#### 페르소나 C: 품질 담당자/플랫폼 운영자

* 저장소 전반의 상태를 모니터링해야 함
* 어떤 팀이 어디서 병목이 발생하는지 알고 싶음
* 원하는 것: 대시보드, 정책 집행, 감사 추적

## 7. 핵심 사용 시나리오

### 7.1 시나리오 1: 현재 Git 상태 이해

개발자는 웹 도구에 저장소와 브랜치를 선택하고, 현재 HEAD, upstream 대비 차이, 미병합 커밋, 작업 브랜치 상태, PR 연결 여부를 시각적으로 확인한다. 도구는 “현재는 main 기준으로 3 commits ahead, 2 commits behind”, “rebase 필요”, “직접 push 금지 브랜치”와 같이 해석된 문장으로 상태를 설명한다.

### 7.2 시나리오 2: merge 충돌 지원

개발자가 PR 병합 또는 로컬 병합 중 충돌을 만나면 도구는 충돌 파일, 충돌 원인, 관련 commit 흐름, 권장 절차를 단계별로 제시한다. 또한 “merge 대신 rebase가 적절한지”, “현재 충돌이 코드 생성물인지 수기 수정 파일인지”를 구분해 안내한다.

### 7.3 시나리오 3: rebase 충돌 지원

개발자가 rebase 중 특정 커밋에서 멈추면 도구는 rebase 대상 커밋, 대상 base branch, 현재 멈춘 이유, 다음 명령 후보, 위험도, 중단 복구 경로를 안내한다.

### 7.4 시나리오 4: cherry-pick/backport 지원

개발자가 릴리스 브랜치에 hotfix를 backport해야 할 때 도구는 cherry-pick 후보 커밋, 의존 커밋, 누락 가능성, 충돌 가능성, 관련 테스트 범위를 분석한다.

### 7.5 시나리오 5: 품질 게이트 기반 PR 생성

개발자가 PR을 열면 도구는 변경 파일 유형, 영향 모듈, CODEOWNERS, 필수 리뷰어, 필수 빌드/테스트/정적분석 항목, 릴리스 영향 여부를 자동 계산해 PR 체크리스트와 상태를 보여준다.

### 7.6 시나리오 6: 운영 대시보드

플랫폼 조직은 조직 전체 저장소의 PR 흐름, 충돌 빈도, merge queue 대기, 리뷰 SLA, 위험도 높은 PR, 정책 위반 추세를 대시보드로 확인한다.

## 8. 제품 원칙

1. Git 내부 개념을 숨기지 않되, raw 용어만 노출하지 않고 해석된 문장으로 설명한다.
2. 실수 예방이 사후 복구보다 우선이다.
3. GitHub Enterprise 기본 기능과 중복 구현하지 않고, 정책/진단/교육/운영 보완에 집중한다.
4. 대규모 임베디드 저장소 성능을 고려해 분석 범위를 단계적으로 수행한다.
5. 조직 규칙은 중앙에서 관리하고 저장소별 예외는 명시적으로 승인한다.

   * REST/GraphQL API 연동
   * Checks API 연동

   * 충돌/복구 가이드 엔진

   * 개발자 포털
   * PR 보조 화면
   * 병합/리베이스/체리픽 지원 화면
   * 학습형 시각화 화면
   * 운영자 대시보드

4. 데이터/관측 레이어

   * 이벤트 저장소

   * 감사 로그
   * 알림 및 리포팅

## 11. 상세 기능 요구사항

### 11.1 공통 정보 구조

도구는 다음 핵심 도메인 모델을 가진다.

* Organization
* Repository
* Team
* Branch
* Pull Request
* Commit
* Review Rule
* Quality Gate
* Risk Signal
* Conflict Case
* Release Branch
* Audit Event

### 11.2 저장소 홈

#### 목적

저장소 단위의 현재 건강 상태를 한눈에 보여준다.

#### 필수 기능

1. 기본 메타데이터 표시

   * 저장소명
   * 기본 브랜치
   * 소유 팀
   * 활성 PR 수
   * 최근 7일 merge 수
   * 오픈된 위험 PR 수
2. 브랜치 상태 요약

   * main/release/develop/feature 패턴별 분포
   * ahead/behind 상태
   * 오래된 브랜치
   * 보호 브랜치 여부
3. 품질 상태 요약

   * 최근 빌드 성공률
   * 필수 체크 누락 PR 수
   * 코드 소유자 승인 누락 PR 수
4. 정책 상태

   * ruleset 적용 여부
   * 예외 승인 저장소 여부
   * merge queue 활성 여부
5. 빠른 진입 링크

   * 위험 PR 보기
   * 병합 대기 보기
   * 정책 위반 보기
   * 최근 충돌 사례 보기

#### 수용 기준

* 저장소 홈 첫 화면 로딩은 3초 이내여야 한다.
* 주요 지표는 5분 이내 데이터 지연을 목표로 한다.

### 11.3 브랜치/히스토리 시각화 화면

#### 목적

HEAD, base branch, ahead/behind, merge 대상, rebase 대상 등을 초급자도 이해하게 만든다.

#### 필수 기능

1. 브랜치 그래프 뷰

   * 커밋 DAG 시각화
   * 현재 선택 브랜치 강조
   * base branch와 비교
   * merge commit, rebase 후 신규 커밋, cherry-pick 후 신규 커밋 구분
2. 상태 설명 패널

   * 현재 HEAD 설명
   * upstream 대비 ahead/behind 설명
   * PR 연결 상태
   * squash/rebase/merge 예상 결과 설명
3. 위험 경고

   * 잘못된 base branch 가능성
   * diverged branch 경고
   * stale branch 경고
4. 작업 추천

   * update branch
   * rebase 권장
   * merge queue 사용 권장
   * release branch 직접 push 금지 안내

#### 고급 기능

* “이 작업을 하면 그래프가 어떻게 바뀌는지” 시뮬레이션
* merge/rebase/cherry-pick 전후 비교 프리뷰

### 11.4 PR 보조 화면

#### 목적

PR의 품질 상태, 리뷰 준비 상태, 위험도를 통합 제시한다.

#### 필수 기능

1. PR 개요

   * 제목, 작성자, 대상 브랜치, 변경 파일 수, 커밋 수
2. 변경 영향 분석

   * 모듈/디렉터리별 변경 영역
   * 빌드 타깃 영향 추정
   * 테스트 범위 추천
   * 생성 파일/설정 파일/핵심 로직 구분
3. 리뷰 준비 상태

   * draft 여부
   * 필수 리뷰어 지정 여부
   * CODEOWNERS 충족 여부
   * 체크리스트 완료 여부
4. 품질 게이트 상태

   * required checks 통과 여부
   * 정적분석 상태
   * 커밋 메시지 규칙
   * 릴리스 영향 플래그
5. 위험도 점수

   * large PR
   * 민감 디렉터리 변경
   * 다중 subsystem 변경
   * 테스트 누락
   * 릴리스 브랜치 대상
   * 과거 충돌 hotspot 파일 포함
6. 권장 조치

   * PR 분할 권고

### 11.12 Backout / Revert 지원

#### 목적

운영 장애, 품질 회귀, 릴리스 차단 이슈가 발생했을 때 변경 이력을 훼손하지 않고 감사 가능한 방식으로 안전하게 되돌리는 표준 워크플로우를 제공한다.

#### 기본 원칙

1. protected branch에서는 reset/force push 기반 되돌리기를 허용하지 않고 revert/backout PR 기반으로만 처리한다.
2. backout은 히스토리 삭제가 아니라 추적 가능한 신규 변경으로 남겨야 한다.
3. release branch 대상 backout은 일반 기능 PR보다 높은 승인 수준과 검증 절차를 요구한다.
4. PR 단위 되돌리기를 기본값으로 하고, 필요한 경우 commit 단위 세부 backout을 지원한다.

#### 주요 사용자 시나리오

1. 이미 main에 병합된 PR에서 장애가 발생해 해당 PR 전체를 되돌리고 싶다.
2. 하나의 PR 중 일부 commit만 문제를 일으켜 특정 commit만 되돌리고 싶다.
3. release branch에 반영된 hotfix가 회귀를 유발해 긴급 backout이 필요하다.
4. backout 후 main, develop, release 간 반영 상태 차이를 추적하고 싶다.

#### 기능 요구사항

1. PR 단위 backout

   * merged PR 선택 후 revert PR 자동 생성
   * 관련 이슈, 원인, 장애 티켓 링크 필수 입력
   * 원본 PR, 원본 작성자, 원본 리뷰어, 영향 브랜치 자동 연결
2. Commit 단위 backout

   * 단일 commit revert
   * 복수 commit 일괄 revert
   * commit 간 의존 관계 경고
   * revert 순서 추천
3. Release backout

   * release 브랜치 대상 전용 backout 요청 화면
   * 긴급도 선택
   * 승인권자 지정
   * 사후 main 반영 여부 추적
4. 영향 분석

   * backout 대상 파일/모듈 표시
   * build target 영향 추정
   * 필요한 회귀 테스트 후보 추천
   * generated file, config, bootloader, kernel, driver, middleware, app 계층별 민감도 표시
5. 충돌 및 위험 경고

   * revert 충돌 가능성 사전 예측
   * 후속 PR 또는 연관 hotfix와 충돌 가능성 경고
   * 부분 backout 시 잔존 리스크 경고
6. 감사 및 추적

   * backout 사유 기록
   * 승인 이력 기록
   * 실제 merge 시각 기록
   * 이후 재적용 여부 추적
7. 브랜치 동기화 추적

   * main에는 backout 되었으나 release에는 남아 있는 변경 탐지
   * release에서만 backout 되었고 main에는 유지되는 변경 탐지
   * 후속 정합성 조치 권고

#### UI/UX 요구사항

1. Backout Center 화면

   * 최근 병합된 PR 목록
   * 위험도 높은 변경 목록
   * backout 가능 상태 표시
   * 브랜치별 backout 필터
2. Backout 상세 화면

   * 원본 PR/commit 개요
   * 되돌릴 변경 요약
   * 예상 영향 범위
   * 승인 요구사항
   * 생성될 revert PR 미리보기
3. 실행 가이드

   * 지금 왜 backout이 필요한지 기록
   * 전체 revert와 부분 revert 중 어떤 선택이 더 안전한지 설명
   * backout 이후 후속 작업 안내
4. 운영 가시성

   * 최근 7일/30일 backout 통계
   * 반복 backout 발생 팀/모듈
   * backout 후 재수정된 영역 표시

#### 정책 요구사항

1. protected branch는 backout도 PR 기반으로만 처리해야 한다.
2. release branch backout은 지정 승인권자의 승인이 필요하다.
3. 고위험 모듈 backout은 CODEOWNERS 재승인이 필요하다.
4. backout PR도 일반 PR과 동일하게 required checks를 통과해야 한다.
5. merge queue 적용 브랜치에서는 backout PR도 queue 진입이 기본이어야 한다.
6. 긴급 backout 예외는 별도 사유와 만료 시간을 기록해야 한다.

#### 수용 기준

1. 사용자는 merged PR에서 3클릭 이내에 backout 요청 생성을 시작할 수 있어야 한다.
2. backout PR 생성 시 원본 PR, 대상 브랜치, 사유, 필수 승인자 정보가 자동 연결되어야 한다.
3. release branch backout 요청은 승인자 미지정 상태로 제출할 수 없어야 한다.
4. backout 이후 운영 대시보드에서 원본 PR과 backout PR의 연결 관계를 조회할 수 있어야 한다.
5. 충돌 가능성 예측이 불가능한 경우에도 최소한 revert PR 생성 가능 여부와 수동 검토 필요 여부를 표시해야 한다.

## 21. 주요 리스크 및 대응

### 리스크 1: GitHub 기본 기능과 중복 투자

대응: GitHub native 기능은 정책 집행의 기반으로 사용하고, 우리 도구는 해석, 가이드, 운영 관측, 조직 표준화에 집중한다.

### 리스크 2: 분석 정확도 부족

대응: 위험도와 영향 범위는 설명 가능한 규칙 기반으로 시작하고, 추후 데이터 기반 고도화로 확장한다.

### 리스크 3: 사용자 거부감

대응: PR 생성과 리뷰 흐름을 방해하지 않고 보조 도구 형태로 시작한다. 차단 규칙은 최소한으로 도입한다.

### 리스크 4: 대규모 저장소 성능 문제

대응: 이벤트 기반 비동기 분석, 캐시, 증분 분석, 파일 수 기반 단계적 분석을 적용한다.

### 리스크 5: 정책 과도화로 개발 속도 저하

대응: 조직 공통 정책과 저장소 예외를 분리하고, 대시보드로 정책 효과를 지속 점검한다.

## 22. 오픈 이슈

1. 조직의 최종 브랜치 전략은 trunk-based, git-flow 변형, release branch 중심 중 무엇으로 표준화할 것인가.
2. SoC 코드베이스가 monorepo인지 multi-repo인지에 따라 영향 분석 모델이 달라진다.
3. 생성 코드와 수기 코드 구분 기준을 어디서 관리할 것인가.
4. 로컬 Git 상태를 어느 수준까지 수집할 것인가.
5. 릴리스 승인 워크플로우를 GitHub 내부로 둘지 외부 승인 시스템과 연동할지 결정이 필요하다.

## 23. 권장 MVP 우선순위

## 25. 정보 구조(IA)

### 25.1 IA 설계 원칙

1. 사용자는 Git 개념이 아니라 현재 해야 할 작업 기준으로 메뉴를 찾을 수 있어야 한다.
2. 개발자용 화면과 운영자용 화면을 분리하되, 동일 데이터 모델을 기반으로 연결되어야 한다.

### 25.6 핵심 화면 정의

#### 홈

개인 중심의 실행 허브 역할을 한다. 사용자는 내 PR, 내 리뷰, 내 위험 알림, 최근 충돌/Backout 현황을 한 번에 보고 바로 액션을 시작할 수 있어야 한다.

#### 저장소 홈

저장소 전체 건강 상태를 보여주는 화면이다. 브랜치 상태, 정책 적용 상태, 최근 이벤트, 품질 지표가 함께 보여야 한다.

#### 브랜치/히스토리 뷰

Git 개념을 시각적으로 이해시키는 핵심 화면이다. HEAD, base branch, ahead/behind, merge/rebase/cherry-pick 결과 예측이 그래프와 설명 문장으로 함께 제공되어야 한다.

#### PR 상세

실무 사용 빈도가 가장 높은 화면이다. 변경 영향 분석, 위험도, 품질 게이트, 리뷰 상태, 관련 빌드와 문서가 통합되어야 한다.

#### 충돌 지원 화면

현재 충돌 상황을 Git 초급자도 이해할 수 있게 설명하는 문제 해결형 화면이다. 상태 해석, 충돌 파일, 다음 명령 후보, 중단/복구 경로가 핵심이다.

#### Backout / Revert Center

운영 안정성 관점의 핵심 화면이다. 되돌릴 대상 선택, 영향 분석, 승인 조건, revert PR 생성, 후속 정합성 추적이 한 흐름으로 이어져야 한다.

#### 정책 관리

플랫폼 팀 중심 화면이다. 조직 표준 정책을 템플릿으로 관리하고 저장소/팀/브랜치에 배포하는 역할을 한다.

#### 운영 대시보드

조직 수준 관측 화면이다. PR 흐름, 충돌, Backout, 병목, 품질 리스크를 추세와 drill-down 구조로 확인해야 한다.

### 25.7 전역 검색 IA

전역 검색은 아래 객체를 통합 검색해야 한다.

* 저장소
* 브랜치
* PR
* commit
* Backout 요청
* 정책 템플릿
* 팀
* 플레이북/문서

검색 결과는 객체 유형별 탭 또는 패싯으로 필터링할 수 있어야 한다.

### 25.8 권한 기반 메뉴 가시성

1. 일반 개발자

   * 홈
   * 저장소
   * Pull Request
   * 충돌 지원
   * Backout / Revert Center
   * 릴리스 운영 일부 조회
   * 학습 센터
2. 리뷰어

   * 일반 개발자 메뉴 + 리뷰 분석 중심 화면
3. 팀 리드

   * 운영 대시보드 일부
   * 정책 조회
   * 예외 승인 관련 화면
4. 플랫폼 운영자

   * 전체 메뉴 접근 가능
5. 시스템 관리자

   * 관리 메뉴 포함 전체 접근 가능

### 25.9 MVP 기준 IA 우선 구현 범위

MVP에서 우선 노출해야 하는 1차 메뉴는 아래와 같다.

```text
1. 홈
2. 저장소
3. Pull Request
4. 충돌 지원
5. Backout / Revert Center
6. 정책 관리
7. 운영 대시보드
8. 학습 센터
```

관리 메뉴와 릴리스 운영의 고급 기능은 2단계 오픈 대상으로 둘 수 있다.

### 25.10 IA 설계 결론

이 제품의 IA는 Git 기능 나열형 구조가 아니라, 개발자가 실제로 겪는 업무 흐름인 상태 확인, PR 생성/리뷰, 충돌 해결, 되돌리기, 릴리스 반영, 정책 준수, 운영 관측의 순서로 조직되어야 한다. 이를 통해 Git 초급자는 필요한 화면을 빠르게 찾고, 운영자는 동일한 정보 구조 안에서 정책과 품질을 통제할 수 있다.

## 26. 화면별 와이어프레임 초안

### 26.1 와이어프레임 설계 원칙

1. 모든 핵심 화면은 상태 요약, 위험 신호, 권장 액션이 한 화면에 보여야 한다.
2. Git raw 정보는 필요 시 확인 가능해야 하지만, 기본 노출은 해석형 문장 중심으로 구성한다.
3. 개발자가 가장 많이 보는 화면은 PR 상세, 브랜치 상세, 충돌 지원, Backout 상세다.
4. 좌측은 탐색, 중앙은 핵심 내용, 우측은 액션/가이드 패턴을 기본 레이아웃으로 사용한다.
5. 임베디드 조직 특성상 영향 모듈, 타깃 빌드, 릴리스 브랜치 상태를 고정 영역으로 반복 노출한다.

### 26.2 공통 레이아웃

```text
┌──────────────────────────────────────────────────────────────┐
│ 상단 바: 조직 / 저장소 / 검색 / 알림 / 사용자 메뉴            │
├───────────────┬───────────────────────────────┬──────────────┤
│ 좌측 메뉴      │ 중앙 메인 콘텐츠               │ 우측 액션 패널 │
│ - 홈          │ - 요약 카드                    │ - 권장 액션    │
│ - 저장소      │ - 상세 분석 영역               │ - 위험 경고    │
│ - PR          │ - 탭/표/그래프/타임라인        │ - 관련 문서    │
│ - 충돌 지원   │                               │ - 승인/실행    │
│ - Backout     │                               │               │
└───────────────┴───────────────────────────────┴──────────────┘
```

```text

├──────────────────────────────────────────────────────────────┤

├───────────────────────┬──────────────────────────────────────┤
│ 내 PR 현황            │ 리뷰 요청함                           │
│ - PR 제목             │ - PR 제목                             │
│ - 상태/위험도         │ - 요청자/기한                         │
│ - 다음 액션           │ - CODEOWNERS 여부                     │
├───────────────────────┼──────────────────────────────────────┤
│ 최근 충돌/Backout     │ 빠른 실행                            │
│ - Merge/Rebase/CP     │ - PR 생성 보조                        │

```

3. 스토리는 역할, 가치, 수용 기준이 분명해야 한다.
4. 플랫폼 팀과 제품 팀이 병렬로 개발할 수 있게 프론트/백/API/분석 작업을 분리한다.

### 27.3 Epic 상세

#### Epic 1. 인증 및 GitHub Enterprise 연동

목표: 사용자가 사내 계정으로 로그인하고 GitHub 조직/저장소 데이터를 안전하게 조회할 수 있게 한다.

주요 스토리

1. 사용자는 SSO로 로그인할 수 있다.
2. 관리자는 GitHub App 설치 상태를 확인할 수 있다.
3. 시스템은 webhook 이벤트를 수신하고 검증할 수 있다.
4. 시스템은 저장소, 브랜치, PR 메타데이터를 동기화할 수 있다.

수용 기준

* 로그인 성공/실패 로그가 남아야 한다.
* GitHub App 연결 실패 시 원인과 재시도 가이드가 보여야 한다.

#### Epic 2. 공통 레이아웃 및 내비게이션

목표: 전역 탐색과 일관된 작업 구조를 제공한다.

주요 스토리

1. 사용자는 좌측 메뉴에서 핵심 기능을 이동할 수 있다.
2. 사용자는 전역 검색으로 저장소, PR, 브랜치, 정책을 찾을 수 있다.
3. 사용자는 최근 본 항목과 알림을 빠르게 열 수 있다.

#### Epic 3. 저장소/브랜치 상태 가시화

목표: 저장소와 브랜치의 현재 상태를 해석형 문장과 그래프로 보여준다.

주요 스토리

1. 개발자는 저장소 홈에서 브랜치/품질/정책 상태를 볼 수 있다.
2. 개발자는 브랜치 상세에서 ahead/behind와 권장 작업을 볼 수 있다.
3. 개발자는 브랜치/히스토리 뷰에서 그래프와 작업 시뮬레이션을 볼 수 있다.

수용 기준

* 선택 브랜치와 base branch 비교가 가능해야 한다.
* 현재 상태를 자연어 해석으로 보여줘야 한다.

#### Epic 4. PR 목록 및 PR 상세

목표: PR의 현재 상태를 한 곳에서 검토할 수 있게 한다.

주요 스토리

1. 개발자는 조건별로 PR을 필터링할 수 있다.
2. 리뷰어는 PR 상세에서 리뷰 준비 상태와 품질 게이트를 볼 수 있다.
3. 팀 리드는 위험 PR을 빠르게 모아볼 수 있다.

#### Epic 5. PR 위험도 및 영향 분석

목표: 큰 PR, 민감 변경, 테스트 누락 등을 조기 경고한다.

주요 스토리

1. 시스템은 PR 위험도를 계산한다.
2. 사용자는 변경 영향 모듈과 타깃 영향을 볼 수 있다.
3. 사용자는 권장 테스트 범위를 볼 수 있다.
4. 시스템은 고위험 PR에 경고를 표시한다.

수용 기준

* 위험도 산식 근거가 설명 가능해야 한다.
* 분석 실패 시 실패 원인을 노출해야 한다.

#### Epic 6. 리뷰어 추천 및 CODEOWNERS 분석

목표: 리뷰 병목을 줄이고 승인 책임을 명확히 한다.

주요 스토리

1. 사용자는 CODEOWNERS 충족 여부를 볼 수 있다.
2. 사용자는 추천 리뷰어 목록을 볼 수 있다.
3. 팀 리드는 리뷰 SLA 위반을 볼 수 있다.

#### Epic 7. 충돌 지원

목표: merge/rebase/cherry-pick 충돌을 이해하고 해결하도록 돕는다.

주요 스토리

1. 사용자는 충돌 유형을 선택하거나 시스템이 자동 감지한 충돌을 볼 수 있다.
2. 사용자는 현재 상태 해석과 다음 액션을 볼 수 있다.
3. 사용자는 충돌 파일 우선순위와 담당 팀 정보를 볼 수 있다.
4. 사용자는 중단/복구 가이드를 볼 수 있다.

수용 기준

* Merge/Rebase/Cherry-pick 세 유형을 구분해서 안내해야 한다.
* HEAD 관련 개념을 그림과 문장으로 설명해야 한다.

#### Epic 8. Backout / Revert Center

목표: 감사 가능한 되돌리기 워크플로우를 제공한다.

주요 스토리

1. 사용자는 merged PR에서 backout 요청을 생성할 수 있다.
2. 사용자는 특정 commit 또는 복수 commit revert를 시작할 수 있다.
3. 사용자는 release backout 요청에 승인자를 지정할 수 있다.
4. 사용자는 revert PR 생성 전 영향 분석과 검증 항목을 볼 수 있다.
5. 운영자는 backout 이력과 정합성 상태를 볼 수 있다.

수용 기준

* revert PR 생성 전 필수 사유 입력이 필요하다.
* release backout은 승인자 없이 생성할 수 없어야 한다.

#### Epic 9. 정책 관리

목표: 조직 표준 규칙을 중앙에서 정의하고 적용한다.

주요 스토리

1. 운영자는 정책 템플릿을 생성/수정할 수 있다.
2. 운영자는 저장소/브랜치에 정책을 적용할 수 있다.
3. 운영자는 예외 정책을 기간 기반으로 승인할 수 있다.
4. 운영자는 감사 로그를 조회할 수 있다.

#### Epic 10. 운영 대시보드

목표: 조직 차원의 품질과 협업 병목을 관측한다.

주요 스토리

1. 운영자는 전사 KPI를 볼 수 있다.
2. 운영자는 팀별/저장소별 drill-down이 가능하다.
3. 운영자는 충돌, Backout, 정책 위반 추세를 볼 수 있다.
4. 운영자는 리포트를 내보낼 수 있다.

#### Epic 11. 알림 및 액션 연계

목표: 위험 신호를 적시에 전달하고 다음 액션으로 연결한다.

주요 스토리

1. 사용자는 위험 PR 알림을 받을 수 있다.
2. 사용자는 backout 승인 요청 알림을 받을 수 있다.
3. 사용자는 알림에서 바로 관련 화면으로 이동할 수 있다.

#### Epic 12. 학습 센터

목표: Git 초급자가 실제 업무 흐름 안에서 개념을 습득하도록 돕는다.

주요 스토리

1. 사용자는 Git 핵심 개념 카드를 볼 수 있다.
2. 사용자는 상황별 플레이북을 볼 수 있다.
3. 사용자는 시각 시뮬레이터를 사용할 수 있다.
